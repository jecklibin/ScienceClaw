# RPA AI 录制统一分段 ai_script 方案设计

## 目标

将当前 AI 录制能力统一为单一模式，满足以下目标：

- 前端只保留一个 AI 录制入口，彻底下线旧的非 Agent 模式
- 不再让模型在 `structured` 原子动作与 `code` 脚本之间来回切换
- 将 AI 的最小规划单位从“单个原子动作”改为“当前页面内的一段 `ai_script`”
- 允许一段脚本内部完成提取、比较、过滤、条件、循环等动态逻辑
- 一旦执行到可能引发页面状态变化的动作，就在该边界处停止本段执行并重新观察页面
- 最终录制结果只保留通过校验的 `ai_script` 步骤，不保留失败尝试和中间错误路径
- 删除不再承担主流程的旧实现，保持代码干净，不保留长期双轨兼容

## 背景与问题

当前 AI 录制存在两个机制级问题。

第一，现有 Agent 模式仍然以“原子动作规划”为核心。即使用户目标本质上要求动态执行，例如“点击 stars 最多的项目”或“如果状态不是完成就反复刷新直到完成”，Agent 也可能先生成一个提取动作，再把提取结果固化成后续固定点击。这会把动态任务错误地退化为“读出一个静态目标，再按固定值操作”。

第二，现有 Agent 模式把“局部子目标完成”与“用户真实目标完成”混在了一起。例如用户目标是点击、打开、下载，但 Agent 只要提取到了一个名称，就可能把它当作阶段性成功；随后即使点击错了，也可能因为“点击了某个项目链接”而被错误判定为任务完成。

这两个问题说明当前机制的拆分维度不对。真正应该作为规划边界的不是“一个原子动作”，而是“页面状态是否即将变化”。

## 核心设计

### 统一模式

系统只保留一种 AI 录制模式：按页面状态边界分段的 `ai_script` 编排模式。

统一后的行为如下：

- 如果用户目标可以在当前页面内闭环完成，则直接生成并执行一个 `ai_script` 段
- 如果脚本执行过程中会触发页面状态变化，则只执行到首次状态变化动作为止
- 每次状态变化后重新抓取页面快照，由 Agent 基于最新页面重新规划下一段
- 最终录制结果由一个或多个 `ai_script` 段组成

这里的“统一模式”并不意味着每次都必须拆成多段。单页内一次完成的任务，本质上就是统一模式下的单段执行。

### 分段边界

分段边界定义为：执行了一个可能导致页面元素、页面状态、可交互上下文发生变化的动作后，当前段必须结束，并重新观察页面。

本方案采用保守判定策略。以下动作执行后，一律视为必须重新观察的边界：

- `click`
- `press`
- `fill` 后提交或回车
- 链接打开
- 按钮提交
- 展开、收起、切换 tab
- 下拉选择
- 触发弹窗、侧边栏、菜单、对话框
- 任何会改变 DOM、可见区域、选中态、分页状态、下载状态的交互

这里不以 URL 是否变化为唯一标准。即使没有跳转到新页面，只要当前页面状态可能发生变化，也必须分段。

### 段内能力

每个 `ai_script` 段允许包含完整的动态逻辑，包括：

- 元素提取
- 多项比较
- 排序、过滤、匹配
- 条件判断
- 循环与轮询
- 在当前页面内定位最终目标

但它有两个硬约束：

1. 如果段内会执行状态变化动作，只能执行到第一次状态变化动作为止
2. 不允许把动态计算出来的结果固化为后续“结构化点击固定值”的另一步

例如“点击当前列表中 stars 最高的项目”在当前页应当生成一段 `state_changing` 脚本：

- 读取当前页可见项目与 stars
- 动态找出最大项
- 直接点击该项
- 段结束

而不是拆成：

- 提取 stars 最高项目名称
- 再点击固定名称

## 执行状态机

统一模式下的执行状态机分为 5 个阶段。

### 1. observe

采集当前页面状态：

- URL
- title
- frame 信息
- 可交互节点
- 内容节点
- 容器结构
- 当前录制上下文

这一步继续复用现有的 frame-aware snapshot 能力。

### 2. plan_segment

模型输入内容包括：

- 用户总目标
- 当前页面快照
- 已提交的录制步骤
- 上一段执行结果
- 最近一次恢复原因

模型输出的是“当前页面上的下一段 `ai_script` 规格”，而不是原子动作。

### 3. execute_segment

运行时执行该段 Python Playwright async 脚本，并采集：

- 标准输出
- 结构化返回值
- 异常信息
- 耗时
- 执行前后页面快照
- 是否检测到状态变化

### 4. reobserve_and_validate

段执行完成后，重新抓取最新页面快照，判断：

- 该段是否达成自己的局部目标
- 页面是否发生了预期变化
- 用户总目标是否已经完成

### 5. continue_or_finish

- 若总目标已完成，则提交当前段并结束
- 若总目标未完成，则进入下一轮 `observe -> plan_segment`
- 若当前段失败或校验失败，则进入恢复流程，但不提交该段

## Segment 规格

每一段 `ai_script` 除了代码本身，还必须包含结构化元数据，作为运行时验证与提交依据。

建议统一为如下结构：

```json
{
  "segment_goal": "在当前趋势列表中动态比较 stars 并点击最高项",
  "segment_kind": "read_only|state_changing",
  "stop_reason": "goal_reached|before_state_change|after_state_change",
  "expected_outcome": {
    "type": "page_state_changed",
    "summary": "进入刚才动态选中的仓库详情页"
  },
  "completion_check": {
    "url_not_same": true,
    "selected_target_key": "selected_repo_name",
    "page_contains_selected_target": true
  },
  "code": "async def run(page): ...",
  "notes": "只执行到首次状态变化动作为止"
}
```

字段说明：

- `segment_goal`
  当前段要完成的局部目标，必须是可验证的动作目标
- `segment_kind`
  `read_only` 表示本段不会触发状态变化，`state_changing` 表示本段预期会触发状态变化
- `stop_reason`
  说明本段应在什么边界结束
- `expected_outcome`
  描述执行后应观察到的页面结果
- `completion_check`
  给运行时提供校验线索，而不是让模型仅凭自然语言自我宣告成功
- `code`
  真正执行的 Python Playwright async 脚本
- `notes`
  可选补充说明，供调试流展示

## 提交规则

最终录制步骤只在以下条件全部满足时才提交：

1. 段脚本执行成功
2. 执行后重新抓取了最新页面快照
3. `completion_check` 校验通过
4. 如果该段是 `state_changing`，则确实发生了该段预期的状态变化
5. 该段没有被后续恢复过程证明为错误尝试

如果任一条件不满足：

- 该段进入调试/恢复流
- 不进入最终步骤列表
- 不参与导出

### 动作型目标的额外约束

如果用户目标是点击、打开、进入、下载、提交、确认等动作型目标，则单纯“提取出一个值”不能视为完成。

例如：

- 用户目标是“点击 stars 最多的项目”
- 若脚本只返回了项目名称，没有完成点击并验证进入目标页面，则该段不能提交

## 恢复机制

统一模式下，失败恢复应分为 3 类。

### 1. 代码无效

典型场景：

- 产出 JavaScript 而非 Python
- 缺少 `async def run(page):`
- 语法错误
- 缩进错误

处理方式：

- 不提交步骤
- 不改变最终录制结果
- 将错误反馈给模型，要求仅重写当前段代码
- 不允许在这一类恢复里偷偷改变当前段目标

### 2. 执行失败

典型场景：

- 超时
- 元素失效
- 点击被遮挡
- 页面已变化导致定位失效

处理方式：

- 先重新观察页面
- 如果页面基本未变，可以在当前段内重试一次或重写一次
- 如果页面实际上已经变了，则不允许基于旧快照盲重试，必须改为重新规划下一段

### 3. 校验失败

典型场景：

- 代码执行成功，但未达到 `expected_outcome`
- 点击发生了，但未进入正确页面
- 提取了错误对象

处理方式：

- 段标记为未提交
- 进入恢复态
- 模型必须基于执行后的最新快照解释未达成原因
- 然后重新规划当前段或下一段

## 前端展示

前端展示应分为两层。

### 调试流 / 对话流

用于展示完整执行过程，包括：

- 当前段目标
- 本段代码
- 执行中状态
- 重新观察结论
- 恢复原因
- 下一段规划

### 最终步骤流

最终步骤面板只显示已提交的 `segment_committed` 步骤。

不应出现在最终步骤流中的内容：

- 失败尝试
- 超时脚本
- 错误提取
- 被后续修正覆盖的旧段
- 仅用于恢复的调试动作

这保证用户最终看到的是“稳定有效的录制结果”，而不是 Agent 的试错过程。

## 后端架构调整

本方案采用“保留底层能力，重写上层规划协议”的策略。

### 保留的能力

- `backend/rpa/generator.py`
  继续负责将最终录制步骤导出为 Playwright 脚本
- `backend/rpa/executor.py`
  继续负责测试执行与结果采集
- `build_page_snapshot` 及 frame-aware snapshot 相关能力
  继续作为统一模式的观察基础
- 现有路由层的 SSE / WebSocket 事件通道
  继续用于前端展示，但事件语义需要替换为 segment 语义

### 需要重写或收缩的部分

- `backend/rpa/assistant.py`
  当前同时承担 structured 规划、code 补救、ReAct 重试、步骤提交等职责，需要重构为 segment orchestrator
- 现有 `structured intent` 主流程
  不再作为 AI 录制主路径，后续应删除其在 AI 录制链中的接线
- `execution_mode=structured|code`
  该二分法退出统一模式主流程
- 当前 `_post_check_subgoal`
  需要升级为通用的 segment validation 机制

### 建议拆分的模块

- `segment_planner`
  输入：用户目标、当前快照、已提交 segments、上一段结果
  输出：下一段 segment spec
- `segment_runner`
  负责执行当前段代码并采集运行信息
- `segment_validator`
  负责判断局部目标是否完成、页面是否发生预期变化、总目标是否完成
- `recording_committer`
  只提交通过校验的 segment
- `attempt_log`
  记录调试信息与恢复轨迹，不进入最终录制步骤

## 事件协议

前端和后端之间的事件建议统一切换为 segment 语义：

- `segment_planned`
- `segment_started`
- `segment_stdout`
- `segment_reobserved`
- `segment_validation_failed`
- `segment_recovering`
- `segment_committed`
- `recording_done`
- `recording_aborted`

最终步骤列表只消费 `segment_committed`。

## 迁移方案

建议按 3 个阶段迁移，避免长时间半新半旧并存。

### 阶段 1：建立统一 segment 主链

- 将 `assistant.py` 改造成统一的 segment orchestrator
- 引入 `segment_planner`、`segment_runner`、`segment_validator`、`recording_committer`
- 让 AI 录制入口只走新的 segment 主链

### 阶段 2：切换前端协议

- 前端只保留一个 AI 录制入口
- 录制页改为展示 segment 调试流与最终步骤流
- 最终步骤面板只显示 committed segments

### 阶段 3：删除旧实现

- 删除旧的非 Agent 模式入口与路由分支
- 删除 AI 录制主路径上的 structured 执行接线
- 删除 `execution_mode=structured|code` 相关提示词、事件、兼容逻辑
- 删除不再使用的旧测试
- 删除不再被引用的辅助函数与模型协议定义

## 代码清理原则

为保持代码干净，本方案明确要求：

- 不保留长期双轨实现
- 不为了“也许以后还会用到”保留旧主流程
- 只保留生成器、执行器、页面快照等底层通用能力
- 所有不再承担主流程职责的旧实现，完成迁移后立即删除

需要重点清理的对象包括：

- 旧非 Agent 模式入口
- AI 录制中的 `structured intent` 规划与执行主链
- `execution_mode` 相关旧协议
- 旧的步骤提交事件与前端分支
- 围绕旧模式编写、但已不再反映目标行为的测试

## 测试策略

至少补充并长期维护以下回归测试。

### 1. 动态比较后点击

用户目标：点击当前列表中 stars 最高的项目

断言：

- 最终只能提交一个 `ai_script` 段
- 不允许退化成“提取名称 + 固定点击名称”
- 必须验证进入的是动态选中的目标页面

### 2. 页面变化后重观察

断言：

- `click`、`press`、提交等动作后必须重新抓快照
- 后续规划只能基于新快照

### 3. 尝试不提交

断言：

- 失败段、超时段、被替代段不能进入最终步骤列表

### 4. 动作型目标不能只读完成

断言：

- 用户目标是点击、打开、下载时，仅提取名称或文本不算完成

### 5. 单页闭环任务

断言：

- 如果任务可以在当前页面内完成，最终只生成一个 `ai_script` 段

### 6. 跨状态变化多段执行

断言：

- 若任务需要多次状态变化，则按边界拆成多个 `ai_script` 段
- 每段结束后都要重新观察并校验

## 风险与约束

### 风险 1：段脚本过长导致执行超时

应对：

- 强约束“首次状态变化即结束本段”
- 对段内轮询设置上限与明确超时分类

### 风险 2：模型仍尝试输出过细原子动作

应对：

- 从提示词与输出协议层面移除 structured 主路径
- 若模型输出不符合 segment spec，直接判为协议错误并要求重写

### 风险 3：校验条件过弱导致误提交

应对：

- 对动作型目标引入更严格的页面变化与目标一致性校验
- 将“模型说完成”与“系统验证完成”彻底分离

## 结论

AI 录制应从“原子动作 ReAct”升级为“按页面状态边界分段的统一 `ai_script` 编排模式”。

该方案可以同时解决以下问题：

- 动态任务被错误固化为静态点击
- Agent 在错误局部路径上自我强化
- 失败尝试污染最终步骤列表
- 非 Agent 与 Agent 双轨共存造成的复杂度和不一致

统一后，系统只有一种主流程：

- 在当前页面内生成一段 `ai_script`
- 执行到首次状态变化为止
- 重新观察页面
- 校验结果
- 继续下一段或结束

这既保留了动态执行能力，也保证了录制结果稳定、可导出、可理解，并为后续清理旧实现提供明确边界。
